# Cravveo 100일 프로젝트 — Phase 1 회고 (Day 001~014)
## 쌩초보가 리눅스를 켜고, 로컬 AI를 돌리고, RAG까지 도달하기까지

> 이 문서는 NotebookLM 등 AI 요약/팟캐스트 생성 도구에 소스로 넣기 위해 정리한 narrative 버전입니다.
> Cravveo Company 100일 AI 파인튜닝 프로젝트의 첫 14일, Phase 1 전체 여정을 시간 순서대로 정리했습니다.

## 프로젝트 소개

Cravveo Company는 기술 초보가 100일 동안 로컬 AI 환경을 직접 구축하고, 파인튜닝까지 도전하는 Build in Public 유튜브 프로젝트입니다. 운영자는 윈도우만 30년 넘게 사용해온 사람으로, 리눅스도 처음, 코딩도 처음, AI 모델을 직접 다루는 것도 처음입니다.

사용 환경은 한성 TFX5556U 노트북 (AMD Ryzen 5 5560U, RAM 16GB, AMD Radeon Vega 내장 그래픽, NVMe SSD 468GB, Linux OS)입니다. 고사양 워크스테이션이 아닌, 일반 보급형 노트북으로 도전한다는 점이 이 프로젝트의 핵심입니다.

Phase 1(Day 001~015)의 목표는 두 가지였습니다. 첫째, 리눅스 환경에서 Python 자동화의 기초를 다지는 것. 둘째, AnythingLLM과 Ollama를 연결해 완전 오프라인 RAG(검색 증강 생성) 시스템을 구축하는 것이었습니다.

## 1막 — 낯선 운영체제와의 첫 만남 (Day 001~003)

### Day 001: 리눅스 세계에 입문하다

처음으로 리눅스 세계에 들어왔습니다. 윈도우만 30년 넘게 사용해온 사람에게는 완전히 새로운 도전이었습니다. "윈도우는 너무 무거울 것 같고, 개발은 윈도우보다 리눅스를 더 많이 쓰는 것 같다"는 생각으로 유튜브를 보며 설치를 진행했습니다.

새로운 운영체제에 적응하는 것은 쉬운 듯 어려운 듯했습니다. 큰 틀은 이해가 가지만, 디테일한 명령어들이 윈도우와 조금씩 달라서 헤매는 날들이었습니다. 터미널을 처음 열었을 때의 그 막막함 — 이것이 100일 여정의 출발점이었습니다.

**한 일**: 터미널 세팅, Python/pip/venv/git 설치
**핵심 발견**: 터미널이 전부다

### Day 002: 가상환경, 알 수 없지만 일단 해보기

가상환경(venv)을 만들었습니다. 그런데 솔직히 말하면, 왜 이걸 해야 하는지 아직도 잘 모릅니다. "다들 해야 한다고 해서" 하고 있을 뿐, "그냥 하면 안 되나?"라는 생각이 계속 듭니다. 서버를 만들어야 해서 그런 게 아닐까 막연히 짐작만 할 뿐입니다. 언젠가는 이해하게 되겠죠.

**한 일**: 작업 폴더 생성 + 가상환경(venv) 구성
**핵심 발견**: 프로젝트마다 환경을 분리해야 한다 (이유는 아직 완전히 이해하지 못함)

### Day 003: Git, 일단 따라 하는 버전 관리

개발자들이 파일을 올려두고, 스타도 받고, 버전 관리도 한다는 이야기를 듣고 "이건 다 해야 하는 거구나" 싶어서 git init부터 첫 커밋, GitHub push까지 따라 했습니다. 다만 정확하고 디테일하게 사용하고 있는 단계는 아직 아닙니다. 일단 무작정 따라 하면서 손에 익히는 중입니다.

**한 일**: git init + 첫 커밋 + GitHub push 성공
**핵심 발견**: 버전 관리 = 실수해도 되는 안전망

## 2막 — 로컬 AI와의 첫 대화, 그리고 한계 (Day 004~007)

### Day 004: Ollama 설치 — 로컬 AI가 진짜 작동한다

여러 로컬 AI 모델 도구가 있었지만 Ollama를 선택했습니다. AI가 추천해줬고, API 연결도 잘 되고, 무엇보다 가장 많이 쓰이고 있다는 점이 결정적이었습니다. gemma4:e2b 모델을 처음 실행하고 대화를 나눈 순간, "로컬 AI가 진짜 된다"는 걸 직접 확인했습니다.

**한 일**: Ollama 설치 + gemma4:e2b 첫 실행 + 첫 대화
**핵심 발견**: 로컬 AI가 진짜 된다

### Day 005: 모델 비교 — 내 노트북엔 너무 무겁다

gemma4:e2b와 qwen3.5:2b 두 모델을 비교했습니다. 그런데 둘 다 지금 노트북에서 쓰기엔 버거웠습니다. 흥미롭게도 gemma4:e2b는 휴대폰 갤럭시 S24에서는 빠릿하게 돌아가는데, 정작 노트북에서는 버벅거렸습니다. AnythingLLM이라는 프로그램 자체가 무거워서 그런 게 아닐까 추측하게 됐습니다.

**한 일**: gemma4:e2b vs qwen3.5:2b 속도/품질 비교
**핵심 발견**: 작은 모델은 빠르지만 할루시네이션이 있다 — 그리고 사양 문제가 처음으로 수면 위로

### Day 006: Python으로 AI를 코드로 부르다

`ask_ai.py`를 작성하며 처음으로 "반자동화"에 성공했습니다. 아직 많이 부족한 수준이지만, 코드로 AI를 호출하는 순간의 쾌감은 분명했습니다. 터미널 채팅과 코드 호출이 완전히 다르다는 걸 깨달은 날이기도 합니다 — 코드로 제어하는 순간 자동화가 비로소 시작됩니다.

**한 일**: requests 라이브러리로 Ollama API 직접 호출
**핵심 발견**: 코드로 AI를 부르는 순간 자동화가 시작된다

### Day 007: 노트북의 한계, 그리고 슬슬 올라오는 욕망

`save_ai.py`를 작성해 15개 질문을 자동으로 처리하고 저장하는 스크립트를 완성했습니다. 그런데 이 과정에서 노트북 사양의 한계가 본격적으로 드러났습니다. 작은 모델은 날짜도 모르고 한자로 답하며, 큰 모델은 아예 작동하지 않는 문제가 생겼습니다. 여러 언어가 뒤섞여 출력되기도 했습니다.

"아~~ 노트북 바꾸고 싶은 욕구와 욕망이 슬슬 올라온다" — 이 시점부터 하드웨어 문제는 이 프로젝트의 숨은 주인공이 되기 시작했습니다.

**한 일**: AI 답변 자동 저장 스크립트 완성, 15개 질문 자동 처리
**핵심 발견**: 작은 모델은 날짜도 모르고, 한자로 답하고, 긴 답변은 잘린다 — 그리고 노트북을 바꾸고 싶다는 욕망의 시작

## 3막 — 나만의 챗봇이 탄생하다 (Day 008~010)

### Day 008: 내가 챗봇을 만들다니

`chat_ai.py`를 작성해 `while True`와 `input()`을 조합한 대화 루프를 구현했습니다. "내가 챗봇을 만들다니" — 신기하면서도, 뭔가 진짜 되고 있다는 느낌을 받은 날이었습니다.

**한 일**: 터미널 대화 루프 구현 — 입력하면 AI가 즉시 답하고 "종료"로 끝나는 구조
**핵심 발견**: while True + input() 조합이 대화의 뼈대

### Day 009: 휘발성에서 영속성으로

지금까지 만든 모든 것은 휘발성이었습니다. 대화가 끝나면 사라졌죠. Day 007의 파일 저장 기술과 Day 008의 대화 루프를 합쳐, 드디어 대화가 파일로 저장되기 시작했습니다. "드디어 저장까지..." — 짧은 한마디지만, 이 프로젝트가 한 단계 성숙해진 순간이었습니다.

**한 일**: 대화 루프에 파일 저장 기능 추가 — `chat_log_XXXXXX.txt` 자동 생성
**핵심 발견**: Day 007(저장) + Day 008(대화 루프) = 영속적인 기록

### Day 010: 완전 자동화, 그 자체

매일 아침 정해진 시간에 AI가 스스로 질문하고, 스스로 답하고, 그 결과를 파일로 저장하는 시스템을 완성했습니다. cron과 Python을 결합한 이 결과물은 "이것이 완전한 자동화다"라는 확신을 준 결정적인 순간이었습니다. Phase 1의 Python 자동화 챕터를 마무리하는 작품이기도 했습니다.

**한 일**: 매일 아침 질문 자동 발송 + 응답 자동 저장 — Cravveo 자동화 도구 완성
**핵심 발견**: cron + Python = 진짜 자동화. 코드 한 번 실행으로 유용한 결과물이 나온다

## 4막 — RAG의 세계로, 그리고 진짜 "내 AI"의 탄생 (Day 011~014)

### Day 011: 윈도우와 리눅스, 다시 만난 차이점

AnythingLLM을 설치하며 윈도우와 리눅스의 또 다른 차이를 느꼈습니다. 윈도우에는 Ollama의 GUI 버전이 있는데, 리눅스에서는 자꾸 터미널로 작업해야 했기 때문에 별도의 GUI가 필요했습니다. 그렇게 AnythingLLM을 설치하게 됐습니다. 클릭만으로 로컬 AI와 대화할 수 있는 GUI의 힘을 처음 체감한 날입니다.

**한 일**: AnythingLLM .AppImage 설치 및 실행, 첫 대화 확인
**핵심 발견**: GUI 없이 터미널만으로 작업하던 한계를 GUI 도구로 보완 — RAG 챕터의 시작

### Day 012: 두 프로그램, 그리고 선택의 어려움

AnythingLLM과 Ollama를 연결했습니다. 두 개의 프로그램이 동시에 돌아가니 노트북이 확연히 느려지는 것을 느꼈습니다. 게다가 모델 종류가 너무 많아서 선택하는 것 자체가 큰 스트레스였습니다. 그래도 워크스페이스에서 첫 대화를 성공시켰습니다.

**한 일**: Settings → AI Provider → Ollama 연동, 워크스페이스 첫 대화
**핵심 발견**: 프로그램이 늘어날수록 사양 부담도 커진다. 모델 선택의 어려움은 생각보다 크다

### Day 013: 기본 AI와 "나의 AI"가 갈라지는 순간

임베딩 모델(nomic-embed-text)을 설치하고 `cravveo_intro.txt` 문서를 업로드해 RAG의 Before/After를 직접 비교했습니다. 문서 없는 AI는 "모른다"고 답했고, 문서를 먹인 AI는 정확히 답했습니다. "드디어 기본 AI와 '나의 AI'가 구별되기 시작한 시작점" — 이 순간이 신기하고 또 의미 있게 다가왔습니다.

**한 일**: 임베딩 모델 설치 → 문서 업로드 → "크라베오 컴퍼니가 뭐야?" Before/After 비교
**핵심 발견**: 문서 없는 AI(모른다) vs 문서 있는 AI(답한다) — RAG가 실제로 작동하는 걸 두 눈으로 확인

### Day 014: 포기할 뻔한 고비, 그리고 결국 성공

문서 17개를 한꺼번에 RAG에 업로드하는 도전이었습니다. 솔직히 이번 편은 포기할 뻔했습니다. 대답도 제대로 나오지 않고, 프로그램이 멈추고, 문서 업로드도 실패하는 일이 반복됐습니다. 모델을 세 번이나 바꿔야 했습니다.

- qwen3.5:2b → "안녕" 한 마디에 2분짜리 Thinking Process를 출력
- llama3.2:3b → 한국어 질문에 베트남어로 답변
- qwen2.5:3b → 드디어 정상 작동

결국 노이즈 문서를 제거하고 모델을 교체하면서 17개 문서 임베딩에 성공했고, "크라베오 컴퍼니는 어떤 회사야?"라는 질문에 정확한 답을 받아냈습니다. 동시에 중요한 한계도 발견했습니다 — "Day 010에서 뭘 했어?"라는 자연어 질문에는 답하지 못했지만, "Python 자동화 cron Ollama 브리핑"이라는 키워드 조합 질문에는 정확히 답했습니다. 작은 모델의 RAG는 아직 키워드 검색에 의존한다는 사실을 직접 확인한 것입니다.

"하아~ 그래도 컴퓨터 사양이 문제였던 거라 어찌어찌 도전해서 결국 성공은 했다" — 가장 힘들었지만, 가장 많이 배운 하루였습니다.

**한 일**: 문서 17개 임베딩 완료, 모델 3번 교체, 노이즈 문서 제거로 정확도 개선
**핵심 발견**: 로컬 AI의 현실 — 작은 모델은 자연어 이해가 아닌 키워드 검색에 의존한다. 그리고 기술보다 하드웨어가 먼저 발목을 잡는다

## Phase 1 핵심 교훈 3가지

### 1. 환경이 전부다
리눅스, Python, Git, Ollama — 이 네 가지 기초가 갖춰지자 그 위에 모든 것이 차곡차곡 쌓이기 시작했습니다. 처음엔 막막했던 터미널이, 지금은 가장 익숙한 도구가 되었습니다.

### 2. 작은 모델의 현실을 알아야 쓸 수 있다
작은 로컬 모델은 빠르지만 할루시네이션을 일으키고, 언어가 섞이며, 자연어 질문보다 키워드 검색에 더 의존합니다. 이 한계를 모르고 쓰면 답답하지만, 알고 쓰면 활용법이 보입니다.

### 3. 사양이 기술보다 먼저 막는다
이번 14일 동안 가장 자주 부딪힌 벽은 알고리즘도, 코드도 아닌 "노트북 사양"이었습니다. 리부팅의 연속, 프로그램이 멈추는 순간들 — 기술적 문제이기 전에 하드웨어의 한계였습니다. 이 문제는 Phase 2(파인튜닝)에서도 계속될 것으로 예상됩니다.

기술적인 문제보다 컴퓨터 사양이 더 큰 벽이었습니다. 리부팅을 반복하면서도 끝까지 해냈습니다. 힘들었지만, 포기하지 않고 14일을 완주했다는 사실 자체가 이번 Phase 1의 가장 큰 성과입니다.
