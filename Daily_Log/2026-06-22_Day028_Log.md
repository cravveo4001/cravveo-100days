# 2026-06-22 Day 028 Log

## 오늘의 목표

학습된 LoRA 가중치를 저장하고, HuggingFace에 업로드하고, 런타임 재시작 후 다시 불러와서 크라베오를 정확히 답하는지 확인한다. "학습 → 저장 → 불러오기" 파이프라인을 완성한다.

## 실행한 작업

- 83개 데이터로 30 epoch 학습 재실행 → loss 0.08 확인
- After 테스트 — 크라베오 답변 정확한지 확인
- `model.save_pretrained("cravveo-lora-adapter")` LoRA 가중치 로컬 저장
- `model.push_to_hub("cravveo/cravveo-llama-lora")` HuggingFace 업로드
- `tokenizer.push_to_hub("cravveo/cravveo-llama-lora")` 토크나이저 업로드
- 런타임 재시작
- 라이브러리 재설치 후 런타임 다시 재시작
- 기본 모델 로드 → `PeftModel.from_pretrained()` 로 LoRA 어댑터 붙이기 → 테스트
- YouTube 업로드 문서 작성

## 확인한 결과

- LoRA 가중치 저장 완료 — `cravveo-lora-adapter/` 폴더 생성
- HuggingFace 업로드 완료 — `cravveo/cravveo-llama-lora`
- 재로드 후 테스트 결과:
  - "크라베오 컴퍼니가 뭐야?" → "IT 프로그램 개발, 인터넷 쇼핑몰, 유튜브 채널 등을 혼자서 운영하면서 수익을 창출하는 1인 기업이야." (정확!)
  - "Build in Public이 뭐야?" → "배우는 과정을 숨기지 않고 공개하면서 진행하는 방식이야." (정확!)
- 파인튜닝 파이프라인 완성: 학습 → 저장 → 업로드 → 불러오기 → 정확한 답변

## 막힌 지점

1. **FastLanguageModel.from_pretrained()로 직접 로드 시 LoRA 미적용**
   - HuggingFace에서 모델을 직접 로드했더니 LoRA 어댑터가 적용되지 않음
   - 엉뚱한 답변 출력

2. **셀 실행 순서 꼬임**
   - 학습 안 된 모델이 업로드될 뻔함

## 해결 방법

1. **기본 모델 + LoRA 어댑터 분리 로드**: 기본 모델(`unsloth/Llama-3.2-1B-Instruct`)을 먼저 로드한 후, `PeftModel.from_pretrained(model, "cravveo/cravveo-llama-lora")`로 어댑터를 따로 붙여야 함
2. **실행 순서 엄수**: "학습 → After 확인 → 저장" 순서 준수

## 오늘의 핵심 발견

- LoRA 어댑터는 기본 모델과 분리해서 저장/로드해야 한다
- `FastLanguageModel.from_pretrained()`로 직접 로드하면 LoRA가 적용 안 된다
- 반드시 기본 모델 로드 → `PeftModel.from_pretrained()`으로 어댑터를 따로 붙여야 한다
- 셀 실행 순서가 곧 결과를 좌우한다 — "학습 → 확인 → 저장" 순서가 중요
