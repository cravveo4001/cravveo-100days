# 2026-07-08 Day 040 Log

## 오늘의 목표

어제 결론(임베딩 검색으로는 정확한 날짜 매칭이 안 됨)에 따라, 날짜→Day 번호를 파이썬 딕셔너리로 직접 매핑하는 룩업 테이블을 구현한다.

## 실행한 작업

- `40일차/date_lookup.py` 작성: `Daily_Log/` 파일명을 스캔해 날짜↔Day번호 딕셔너리 생성, 질문에서 날짜/Day번호를 정규식으로 추출, 해당 파일을 찾아 Ollama(cravveo:v4)에 컨텍스트로 전달
- Day 039 테스트 질문 10개로 파일 매칭만 먼저 검증 → **10/10 정확히 매칭** (임베딩 검색과 달리 100% 정확)
- 실제 답변 생성 단계에서 두 가지 문제 추가 발견:
  1. Modelfile-v4의 TEMPLATE과 파이썬 프롬프트가 "질문:/답변:"을 이중으로 감싸 모델이 학습 데이터의 다른 질문/답변들을 무관하게 나열함 → 프롬프트 수정으로 해결
  2. 수정 후에도 모델이 "다음 날은 뭐 할 거야?" 학습 패턴을 스스로 이어가며, 질문 하나에 답하고 끝나지 않음 → num_predict 축소 + 컨텍스트 트리밍으로 완화했으나 완전히 해결은 안 됨
- 학습 데이터(`20일차/cravveo_identity.jsonl`)에 "쇼핑몰"이라는 잘못된 정체성 정보가 박혀있고, 이게 여러 답변에 반복적으로 튀어나오는 원인임을 확인
- `2026-07-01_Special_Roadmap_Design.md`(300일 로드맵)를 다시 확인 — Part 1(Day 001~100)의 실제 목표는 "기존 툴 그대로 쓰면서 불편함 발견 + 기록"이고, 직접 개발로 완전히 해결하는 건 Part 3(Day 201~300) 몫이라는 걸 재확인
- `불편함_기록.md` 신설 — 오늘까지 발견한 모든 도구/모델 한계를 기록

## 확인한 결과

- **날짜/Day 조회는 코드로 100% 해결 가능함을 검증.** 임베딩 검색과 달리 실패 케이스가 없음
- **답변 생성(요약)은 여전히 소형 모델의 불안정성에 좌우됨.** 완전히 고치려 하지 않고 불편함으로 기록만 하기로 결정
- 학습 데이터의 정체성 오류("쇼핑몰")도 같은 원인 계열 — 프롬프트로 못 고치고 재학습이 필요한 문제

## 막힌 지점

프롬프트/파라미터 튜닝을 몇 번 더 해도 모델이 "질문 하나에 깔끔하게 답하고 멈추기"를 안정적으로 못 함.

## 해결 방법

Part 1의 목적(기존 툴 익히기 + 불편함 기록)에 맞게, 이 문제를 오늘 완전히 뿌리 뽑으려 하지 않고 `불편함_기록.md`에 남겨두기로 함. 대신 원인은 명확히 문서화(이중 템플릿 버그, 학습 데이터의 "다음 날" 패턴 습관).
