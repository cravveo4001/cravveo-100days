## Day 038 — AnythingLLM 워크스페이스 설정 3중 버그

- 채팅 모드가 기본값 "에이전트"라서, 소형 로컬 모델(`llama3.2:3b`)이 도구 호출을 못 해 RAG 검색 자체가 실행 안 됨
- Search Preference "Default"가 관련성 낮은 문서를 가져와 환각 유발
- 최대 문맥 조각을 과도하게 늘리면(100) 모델 컨텍스트 용량을 초과해 정답 문서가 잘려나감
- **불편함**: GUI 기본값들이 소형 로컬 모델 환경에 안 맞게 세팅되어 있고, 이걸 알아채려면 앱 내부 DB(SQLite)를 직접 열어봐야 했음

## Day 039 — 문서 힌트 보강으로 검색 정확도 개선 안 됨

- 로그 파일에 "이 기록은 2026-06-27, Day 032에 해당합니다" 같은 명시적 문장을 추가해도, 재임베딩까지 정상 확인됐음에도 검색 결과가 바뀌지 않음
- **불편함**: 임베딩 기반 벡터 검색은 문서 구조가 거의 동일한 경우(날짜만 다른 37개 로그) 정확한 숫자/날짜 매칭을 구조적으로 못 함. AnythingLLM에는 "정확히 일치하는 것 찾기"(exact match / metadata filter) 옵션이 없음

## Day 039 — q4_0 양자화로 모델 품질 붕괴

- Colab에서 `quantization_method="q4_0"`으로 변환한 모델이 문맥과 무관한 헛소리를 생성
- `q4_K_M`(K-quant)으로 바꾸니 크게 개선됨
- **불편함**: 어떤 양자화 방식을 쓸지 미리 알기 어렵고, 실제로 만들어서 대화해보기 전까지는 품질 저하를 예측할 수 없음

## Day 040 — 날짜/Day 번호 기반 질문에 RAG가 정확히 답 못 함

- AnythingLLM의 벡터 검색으로는 "2026-05-23일에 뭐했어?" 같은 정확한 날짜 질문에 계속 틀린 문서를 가져옴
- 파이썬으로 날짜↔파일 딕셔너리를 직접 만들어 우회(`40일차/date_lookup.py`) — 파일 찾기는 100% 정확했지만, 그 내용을 Ollama에 넘겨 답을 생성하는 과정에서 새 문제 발견:
  - Modelfile의 TEMPLATE과 파이썬 프롬프트가 "질문:/답변:"을 이중으로 감싸 모델이 혼란스러워함 (수정함)
  - 수정 후에도 모델이 학습 데이터의 "다음 날은 뭐 할 거야?" 패턴을 스스로 이어가며, 질문 하나에 답하고 끝나지 않고 다음 질문·답변까지 계속 생성함
- **불편함**: 정확한 조회(lookup)까지는 코드로 100% 해결되지만, 그 결과를 자연어로 "요약"하는 마지막 단계는 여전히 소형 파인튜닝 모델의 불안정성에 좌우됨. AI 요약 없이 원문을 그대로 보여주는 게 더 나을 수도 있음

## Day 040 — 학습 데이터에 잘못된 정체성 정보("쇼핑몰")가 박혀있음

- `20일차/cravveo_identity.jsonl`에 "크라베오 컴퍼니가 뭐야?" → "...인터넷 쇼핑몰..." 이라고 되어있는데, 실제로 쇼핑몰은 없음
- 이 데이터로 v2/v3/v4까지 계속 재학습해와서, 모델이 여러 답변에서 반복적으로 "쇼핑몰"을 언급함
- **불편함**: 파인튜닝 데이터의 오류는 프롬프트로 못 고치고, 데이터를 고친 뒤 재학습해야만 사라짐. 초기(Day 020)에 넣은 정체성 정보를 나중에 고치려면 재학습 비용이 매번 발생함

## Day 041 — "아침 브리핑 자동화" 파이프라인이 정체성 학습 데이터를 오염시킴

- Day 010~023에 만든 `briefing_pipeline.py`는 원래 "매일 아침 AI한테 질문 몇 개 보내고 저장"하는 용도였는데, 그 결과물이 크라베오 정체성 학습 데이터셋(`cravveo_briefing_dataset.jsonl`)에 그대로 append되고 HuggingFace에 자동 업로드되는 구조였음
- 그래서 92개 데이터 중 약 40개가 "AI 유튜버 마케팅 전략", "저작권 법적 대응", "자산 증식 전략" 같은 크라베오랑 전혀 무관한 일반 AI 잡담으로 채워짐 (일부는 한자가 섞이고 문법이 깨진 저품질 텍스트)
- **불편함**: "브리핑용 일반 Q&A 로그"와 "정체성 학습용 데이터"가 처음부터 파일/파이프라인이 분리되어 있지 않아서, 자동화 스크립트 하나가 무심코 학습 데이터를 오염시켰다. 용도가 다른 데이터는 애초에 저장 위치부터 분리해야 했음
